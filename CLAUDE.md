# Struq

## Overview

Struq is a web app for musicians to quickly map out song structures (Intro → A → B → Chorus → Solo → Outro) and reference them during practice or performance.

**URL:** struq.focuswave.cc

## Tech Stack

- **Framework:** TanStack Start (React) + Vite
- **Database:** Cloudflare D1 (SQLite) + Drizzle ORM
- **Deployment:** Cloudflare Workers
- **Linter / Formatter:** Biome
- **Testing:** Vitest (unit) + Playwright (E2E)
- **Auth:** Google OAuth
- **AI:** Google Gemini Flash API (song structure generation)
- **Styling:** Tailwind CSS v4

## Source Structure

```
src/
├── routes/          # TanStack file-based routing (routeTree.gen.ts is auto-generated — do not edit)
├── auth/            # Session, OAuth, server functions
├── db/              # Drizzle schema + getDb() factory
├── songs/           # Song server-fns, performance hooks (use-section-timer, click-preference), constants, components
├── setlists/        # Setlist server-fns
├── i18n/            # Translation keys + I18nProvider
├── server/          # Shared server helpers (requireUser, now)
└── lib/             # Utilities (logger etc.)
```

## Routes

```
/                   → Landing / redirect to /setlists
/login              → Google OAuth login
/setlists           → Setlist list view
/setlists/:id       → Setlist detail (songs in order)
/setlists/new       → New setlist creation
/songs              → Song list
/songs/new          → New song (with AI generate option)
/songs/:id          → Song edit view
/songs/:id/perform  → Performance view (fullscreen, dark)
/settings           → User settings (locale, account)
```


## UI/UX Guidelines

- Minimal, calm aesthetic; keep animations subtle
- Sections color-coded consistently across views
- Touch-first design (large tap targets)
- Performance view optimized for iPad landscape (fullscreen, dark)
- Edit/list views optimized for mobile (iPhone)

## Environment Variables

For local development, create `.dev.vars` (NOT `.env`) in the project root.
Wrangler automatically injects this file into the Worker context during `pnpm dev`.

```
GOOGLE_CLIENT_ID=xxx
GOOGLE_CLIENT_SECRET=xxx
GEMINI_API_KEY=xxx
SESSION_SECRET=xxx
```

For production, set via `wrangler secret put`.

## Gotchas

- E2E auth state is stored in `e2e/.auth/user.json` (gitignored); global setup hits `/api/auth/test-login`
- Server functions use `createServerFn()` + `.inputValidator(ZodSchema)` + `requireUser()` — see any `*server-fns.ts` for reference
- `src/routeTree.gen.ts` is auto-generated by TanStack Start — do not edit manually
- Local env vars go in `.dev.vars` (NOT `.env`) — Wrangler injects this automatically during `pnpm dev`
- D1 binding name is `DB` (defined in `wrangler.jsonc`)
- `@/*` resolves to `./src/*`
- `SESSION_SECRET` must be ≥32 characters
- Soft deletes via `deleted_at`; exception: SetlistSongs uses hard delete
- D1 has a 100 bound-parameter limit per statement; section inserts batch at 10 rows (9 columns × 10 = 90 params) to stay under the limit
- Sections `label` field is only used for `custom` type — fixed types derive their label from locale at render time
- Unit tests cover pure functions only (schemas, validation, constants); server functions depend on D1 + session and are tested via E2E only
- Server functions that operate on both a setlist and songs must verify ownership of **both** (see `addSongToSetlist` for the reference pattern)
